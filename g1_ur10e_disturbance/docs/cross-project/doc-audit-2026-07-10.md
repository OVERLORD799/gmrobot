# GMDisturb 文档可靠性审计

> 结论：**文档不可靠。文档描述的是 v1 设计（概率状态机），代码实现的是 v2（距离门控调速）。**
> 所有文档应视为**设计稿**而非**规格说明**，以代码为真相源。

---

## 核心矛盾：两套完全不同的控制器架构

### 文档描述的 (v1 — 设计稿)

```
WANDER ←→ APPROACH_ARM → RETREAT
  │            │              │
  │   15%概率切换       接触力>20N触发
  │   持续2-3秒         退后几步
  │   dist映射速度      回到WANDER
  │   arm伸出
```

来源: ARCHITECTURE.md §约束随机游走, DATA_FLOW.md Phase B, INTERFACES.md §ConstrainedWanderController, SCENARIOS.md §constrained_wander

### 代码实际实现的 (v2 — 真相)

```
AGGRESSIVE (>0.30m)  →  全速随机游走 (vy=0, 确定性schedule)
MODERATE (0.15-0.30m) →  降速 + 偏置远离UR10e
CAUTIOUS (<0.15m)     →  按距离紧急度撤退 (0.20→0.50 m/s)
STUCK (卡住检测)      →  接触力方向撤退 (force-based, fallback随机)
```

来源: [g1_disturbance_controller.py:199-216](../../g1_disturbance_controller.py#L199-L216)

## 影响评估

### 对前次对抗性审查的影响

上次审查的 H1 (vy=0) 发现需要重新定性：

- **H1 不再是代码缺陷**。vy=0 是工程决策（行走策略仅训练前向行走，横向运动会 destabilize），文档的 vy∈[-0.2,0.2] 是设计假设未验证。
- 修复方向从"恢复 vy 随机"改为"文档解释为什么 vy=0"。

其余发现 (H2 负距离, H3 碰撞过滤, M3-M9) 不受影响——那些是真正的代码问题。

### 对新开发者的影响

如果新人按 ARCHITECTURE.md 理解系统：
- 会寻找不存在的概率状态机和 APPROACH_ARM 逻辑
- 会试图配置不存在的 YAML 字段
- 会期望不存在的 `get_arm_targets()` 方法
- 会以为 G1 在 y 轴上随机游走

**建议在每份文档开头添加时效性声明**：

```
⚠️ 本文档为设计阶段产物（2026-06）。实际实现可能与描述不同。
以源代码为准。最后同步: [从未同步]。
```
